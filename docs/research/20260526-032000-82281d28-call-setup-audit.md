# 82281D28 Call Setup Audit

## Summary

The mapped `82281D28:8228233C-82282370` hot block is not a safe local
store-elision patch target. The parent stores in this block are all live into
the callee under the current A64 guest-call ABI:

- the `0x826BF770` helper call needs `r3`, `r4`, `r5`, and `lr`;
- the recursive `0x82281D28` call needs `r3-r10` and `lr`;
- strict local promotion found zero safe wins for `r3-r10/lr`.

The performance issue is therefore structural: generated A64 guest calls pass
guest arguments through `PPCContext` memory, and callees immediately reload
those arguments from memory. A useful next lane is a generic default-off
guest-call argument handoff / fast-entry audit, not another one-span
`82281D28` behavior patch.

This is not speed proof and does not change runtime behavior.

## Source-Level ABI Finding

`A64Emitter::Call` currently passes only the guest return address in `x0` to
guest machine code. Direct guest calls still expect arguments and volatile state
in `PPCContext`:

```text
parent HIR stores r3-r10/lr -> PPCContext
A64Emitter::Call loads target machine_code
x0 = StackLayout::GUEST_CALL_RET_ADDR
blr target
callee prolog loads r3-r10/lr from PPCContext
```

`CONTEXT_BARRIER` is a no-op in A64 codegen, but it still marks conservative
boundaries for analysis. The actual cost here is not the barrier instruction;
it is the current memory-based guest function ABI and the store/reload traffic
around hot recursive calls.

## Decision

Closed for immediate behavior work:

- local store elision in `82281D28:8228233C-82282370`;
- another narrow first-`CONTEXT_BARRIER` audit;
- another unchanged call-edge capture for this block;
- a one-off caller inline of `826BF770` from this block.

Open structural lane:

- design a default-off, counter-only guest-call argument handoff audit for A64
  direct guest calls.

The audit should be generic, not Blue-Dragon-only at first. It should count:

- direct guest calls where the parent stores `r3-r10` and/or `lr` immediately
  before the call;
- callee first-use loads of those same fields before overwrite;
- tail vs non-tail calls;
- compiled direct calls vs indirection/resolve path;
- extern/helper/debug/exception visibility blockers;
- stackpoint/longjmp synchronization requirements;
- estimated stores and reloads avoidable if a fast-entry variant accepted
  GPR argument payloads out-of-band.

Only after that counter proves broad route volume should the code move toward a
normal-entry-compatible fast-entry variant. The minimum viable design would
need normal fallback, no hidden guest-visible state, explicit invalidation, and
correct behavior for exceptions, debug stops, stack synchronization, and
extern/helper calls.
