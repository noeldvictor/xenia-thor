## Infinite Undiscovery: A64 stackpoint overflow

Abort message: `Overflowed stackpoints! Please report this error for this title
to Xenia developers.` ->
`A64Emitter::HandleStackpointOverflowError` (`a64_emitter.cc:6109`), guest-code
context (frame #03 in the JIT code cache).

- Capacity is the cvar `a64_max_stackpoints`, default **65536**
  (`a64_backend.cc:61`), allocated at backend init.
- 65536 is a huge call depth, so overflow likely indicates a **push/pop
  imbalance (stackpoint leak)** in the A64 backend, not legitimate recursion.
  Bumping the cap would only delay it; disabling
  `a64_enable_host_guest_stack_synchronization` (which no-ops PushStackpoint)
  removes the mechanism but risks longjmp/unwind correctness. Neither is a clean
  fix -> this is a deeper ARM64-backend lane (audit PushStackpoint/PopStackpoint
  balance), not a quick win. Documented for a dedicated future iteration.
- `a64_max_stackpoints` is not currently launch-wired (would need allow-list +
  rebuild to A/B a higher value).

## High-leverage pattern: "boots-but-black"

3 of 4 RPGs (Blue Dragon, Lost Odyssey, MagnaCarta 2) plus the known
Sylpheed/Banjo cases all **boot and run guest code (VdSwap advancing) but render a
black frame**. This is a recurring present / render-target / frontbuffer class.
Like the Gears stride fix unblocked the whole Gears cluster, a single fix to this
black-frame path could unblock several titles at once -> strongest next lane.
Lost Odyssey is a good probe subject (uncapped ~166 fps, lots of guest progress).
