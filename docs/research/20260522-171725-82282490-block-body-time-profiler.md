# 82282490 Block Body-Time Profiler

Date: 2026-05-22 17:17 EDT

## Goal

Replace the noisy `82282490` entry-count guess with generated-code body-time
attribution per HIR block. This is a profiler/tooling slice, not a speed patch.
The new profiler is default-off and function-filtered so normal Blue Dragon
routes do not pay the cost.

## Implementation

New default-off cvar:

- `arm64_speed_profile_block_body_time=false`

Launch helper:

- `tools/thor/thor_xenia_debug.ps1 -Arm64SpeedProfileBlockBodyTime true`

Android extra:

- `arm64_speed_profile_block_body_time`

Runtime behavior:

- The profiler only arms when the A64 speed profiler is on, the block filter is
  non-empty, and the filter matches the generated function.
- It reuses the existing delayed body-time gate, so the run can skip loading
  churn and start after `arm64_speed_profile_body_time_after_ms`.
- The emitter keeps two stack slots for the currently active block:
  `CNTVCT_EL0` start tick and atomic counter pointer.
- Each HIR block label closes the previous active block and starts the next one.
  Function exits close any active block before normal body-time accounting.
- `tools/thor/thor_hir_block_mix_report.ps1` now parses both entry-count block
  rows and new block body-time rows, then emits a `Dynamic Body-Time Blocks
  With HIR Mix` section.

This is approximate block attribution: branch/helper/call work is charged to
the active block until the next HIR block label or function exit. That is the
right bias for choosing codegen targets because it follows generated-code wall
time, but it is still too expensive for normal captures.

## Result

The entry-count story was misleading. `822824F0` is hot by entries and carries
the vector shape, but it is not the body-time wall.

Top `82282490` block body-time totals from the 17:09 capture:

| Guest block | Body total | Peak delta | Peak ticks/entry | HIR read |
| --- | ---: | ---: | ---: | --- |
| `822825E0` | `34726883` | `14525259` | `61` | Small branch plus recursive `bl 0x82282490`; context `2/6`. |
| `822825C8` | `3216407` | `1041116` | `500` | Low-entry but expensive call block, `bl 0x8227FEE8`; context `4/6`. |

Function-level delayed body-time still showed `82282490` as the generated-code
body wall after activation:

- `17:12:14`: `body_ticks_delta=20199870`, `entries_delta=74200`,
  `ticks_per_entry=272`
- `17:12:29`: `body_ticks_delta=15717284`, `entries_delta=60844`,

## Decision

Do not start the `822824F0` `stvewx` peephole yet. It is mechanically
interesting but not the current body-time wall.

The next experiment should target the `822825E0` recursion/call path or the
`822825C8` low-count high-cost call path. The first safe next slice is an
offline caller/callee audit:

1. Explain why `822825E0` recursively calls `82282490` so often.
2. Inspect callee `8227FEE8` from `822825C8` and decide whether the high
   ticks-per-entry cost is generated code, helper boundary, or callee body.
3. Only after that choose a default-off codegen/HLE experiment.
