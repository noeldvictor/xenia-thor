## Source Review

`src/xenia/cpu/backend/a64/a64_emitter.cc` implements
`TryEmitGprLrHelperCall`. For `__savegprlr_*` / `__restgprlr_*`, it reads and
writes `r[1]`, `r[first..31]`, `r[12]`, and `lr` / return state. It calls
`ForgetFpcrMode()`, but it does not read or write `PPCContext::f`,
`PPCContext::v`, or `PPCContext::fpscr`.

`src/xenia/cpu/xex_module.cc` also detects GPR/LR helpers separately from FPR
and VMX helpers. This supports treating `__savegprlr_28` and
`__restgprlr_28` as FPR-safe for a narrow carrier audit.

The two unknown calls in the HIR are not helpers:

```text
82287854 bl 0x821CE028
82287ED4 bl 0x821CE028
```

Previous route evidence already shows `821CE028` is a real child function in
this opening route, not a trivial helper. Do not carry `f[1]` across those
calls without a separate clobber/use proof.

## Decision

- Do not implement a speed patch from this audit alone.
- A helper-whitelist `f[1]` carrier can only cover 2 of 10 static `f[1]` loads
  in `82287788` under the current evidence.
- The remaining 8 loads cross at least one unknown `821CE028` call and must
  remain blocked unless a focused callee proof shows the value is preserved or
  explicitly forwarded.
- Keep `fpscr` out of this lane. It is mutable in `82287788` and remains
  high-risk.

## Next

The next useful slice should be one of:

- a default-off runtime counter that measures dynamic `f[1]`
  helper-whitelist hits versus unknown-call-blocked hits for `82287788`; or
- a focused `821CE028` clobber/use audit proving whether `f[1]` is preserved,
  consumed, or overwritten across the two child calls.

Only move to a quiet Thor A/B if the runtime counter shows enough dynamic
payoff to justify replacing more than the first two static loads.
