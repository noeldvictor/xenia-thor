## Source Implication

This proves the current `82281D28` wall is scalar guest memory addressing around
the guest stack, not VMX/vector/GPU work. It also means a direct argument-handoff
patch is not the right shape: the hot work is still ordinary guest memory
`LOAD_OFFSET` / `STORE_OFFSET`, with PPC-visible endian and 32-bit guest address
semantics.

The current source path for offset memory goes through:

- `src/xenia/cpu/backend/a64/a64_seq_util.h`
  - `ComputeMemoryAddress`
  - `AddGuestMemoryOffset`
- `src/xenia/cpu/backend/a64/a64_seq_memory.cc`
  - `OPCODE_LOAD_OFFSET`
  - `OPCODE_STORE_OFFSET`

That path must preserve 32-bit guest address wrapping and the `0xE0000000`
granularity adjustment before host membase is applied. Any optimization must
prove it does not silently turn guest 32-bit wrap into host 64-bit pointer math.

## Decision

Do not patch generated behavior and do not run a quiet speed A/B from this
offline audit alone.

The evidence justifies a source-level A64 memory-lowering feasibility audit for
`LOAD_OFFSET` / `STORE_OFFSET`, especially `r1` plus small constant forms. The
next slice should inspect whether the existing `ComputeMemoryAddress` /
`AddGuestMemoryOffset` sequence is already the legal instruction floor, or
whether a generic guarded lowering can reduce instructions while preserving
guest wrap, membase, byte-swap, MMIO/exception visibility, and fallback behavior.

If that source audit cannot produce a reusable backend rule, close the current
fastmem/addressing lane for `82281D28` and move to a different structural lane.
