# 82287788 FPSCR CFG Writeback Plan

Date: 2026-05-24 18:49 EDT

Branch: `master`

## Goal

Turn the prior `fpscr` dirty-cache audit into an implementation contract for a
safe no-behavior-change counter patch. This slice is offline-only and does not
change generated code.

The broader context is the hot Blue Dragon opening-route edge:

```text
82282490:82282598 -> 82287788
```

That edge still needs variant storage before any payload can be materialized,
and `fpscr` remains the large but risky mutable payload candidate.

## New Tool

Added:

```text
tools/thor/thor_hir_fpscr_cfg_writeback_plan.ps1
```

The tool wraps `tools/thor/thor_hir_fpscr_dirty_cache_audit.ps1`, keeps the HIR
parser in one place, and emits the exact counter-only contract for a future
`fpscr` CFG writeback skeleton:

- static `fpscr` load/store sites;
- dirty store-to-next-load transition classes;
- CFG boundary kinds that must carry dirty state;
- required call writeback PCs;
- a no-behavior-change decision and next-patch rule.

## Interpretation

The opportunity is still real but not patchable as a local peephole:

- there are zero strict same-window reload wins;
- twenty-four transitions need dirty `fpscr` to survive CFG joins;
- two transitions cross external call visibility;
- behavior cannot change until exact writeback is proven at
  `82287ED4`, `82287EDC`, `82287EE4`, and `82288220`.

The safest next runtime edit is therefore not a speed A/B. It is a default-off
counter-only skeleton that proves the generated-code patchpoint can see every
dirty store, CFG carry, and required call writeback while leaving normal entry
behavior and payload materialization unchanged.

## Decision

Do not patch `fpscr` behavior and do not run a quiet FPS A/B yet.

  `82282490:82282598 -> 82287788`.
