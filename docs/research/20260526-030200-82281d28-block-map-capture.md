# 82281D28 Block Map Capture

## Summary

The HIR block-profile mapper lane is now resolved for the current
`82281D28` route capture. The first capture proved the new stamps were deployed
but also exposed a log transport issue: the Thor logcat buffer was still 256 KiB
and only retained the tail of the one-function OptHIR dump. After increasing
the device logcat buffers to 64 MiB, the repeated capture produced
metadata-backed and `hir_block_*` backed mappings for all active
`82281D28` profile rows.

This is not speed proof. It is tooling proof that weighted `82281D28` HIR
analysis can now use explicit block metadata instead of unsafe ordinal
fallback.

The mapped hot block is scalar call/stack/context setup, not VMX or GPU work.
It computes stack argument addresses from `r1`, stores `r11`, sets up
`r3/r4/r5`, calls `0x826BF770`, reloads stack arguments into `r5-r10`, and
then recursively calls `0x82281D28`.

Prior notes classify `0x826BF770` as a byte-copy / memcpy-shaped helper, and
this route already has `arm64_blue_dragon_memcpy_fastpath=true`. The current
candidate wall is the caller-side stack/context/call-boundary setup around that
helper plus the recursive call, not broad vector lowering and not broad Vulkan.

## Decision

- Do not rerun the unchanged HIR block-profile stamp capture.
- Do not use `-AllowOrdinalFallback` for behavior decisions.
- Treat the mapper lane as complete for this route. Use metadata and
  `hir_block_*` fields for weighted `82281D28` HIR analysis.
- Next useful slice is a targeted offline/source audit of
  `82281D28:8228233C-82282370` call setup:
  - classify why `r3-r10` and `lr` are flushed to PPC context around
    `0x826BF770` and the recursive `0x82281D28` call;
  - verify which stores are required by callee ABI, exception visibility,
    helper behavior, or return handling;
  - compare this shape against existing
    `a64_enable_host_guest_stack_synchronization` and
    `arm64_blue_dragon_memcpy_fastpath` behavior;
  - decide whether a reusable A64 helper ABI, direct-call, stack-argument, or
    caller-local rule exists before any behavior patch.

If the audit cannot produce a reusable backend rule, close this local
call/setup lane and move to the next structural lane rather than adding another
one-PC cvar.
