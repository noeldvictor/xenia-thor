# 82287788 FPSCR Dirty-Cache Audit

Date: 2026-05-24 11:46 EDT

## Question

After the `f[1]` carrier and pair-entry lanes missed speed proof, `fpscr`
looked tempting because the `82282490 -> 82287788` forwarding plan gave it a
large static upper bound:

- `fpscr` loads in `82287788`: `26`
- `fpscr` stores in `82287788`: `26`
- hot parent edge calls: `1691272`
- dynamic access upper: `43973072` loads and `43973072` stores

This audit asks whether a dirty `fpscr` cache has cheap, exact writeback points,
or whether it requires a full CFG/visibility model before any behavior patch.

## Tooling

Added:

- `tools/thor/thor_hir_fpscr_dirty_cache_audit.ps1`

The tool parses a filtered HIR dump, finds `PPCContext::fpscr` accesses
(`+2628`), pairs each load/store update, then classifies each store-to-next-load
transition as:

- `strict_same_window`
- `cfg_phi_or_stack_carrier_required`
- `external_visibility_flush_required`
- `exit_flush_required`

It also reports call PCs that would need a dirty writeback before external
visibility.

## Decision

Do not patch `fpscr` forwarding or dirty caching next.

The opportunity is large on paper, but all useful savings are cross-boundary and
mutable. A safe implementation needs a CFG-aware carrier/PHI or stack-slot model
with exact writeback before calls, returns/exits, exceptions, and any helper or
reader that observes `PPCContext::fpscr`.

Next useful speed lane:

- return to broader `8228252C-822825C4` state/vector/FPR work with body-backed
  local spans; or
- design a generic CFG-aware dirty state-carrier audit first, not a Blue
  Dragon-only `fpscr` behavior shortcut.
