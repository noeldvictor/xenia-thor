## Candidate Shape

The safe behavior shape is a default-off callee-local fpscr dirty stack slot:

```text
candidate=callee_local_fpscr_dirty_stack_slot
normal_entry_behavior=unchanged
parent_to_callee_payload=current_direct_call_abi_blocks_extra_payload
seed_source=first_82287788_context_load_or_normal_context_fallback
dirty_state=callee_local_dirty_flag
load_rule=serve_from_carrier_only_after_seed
store_rule=update_carrier_and_mark_dirty
writeback_rule=flush_dirty_before_cfg_external_visibility_and_required_call_pcs
required_call_pcs=82287ED4,82287EDC,82287EE4,82288220
fallback_rule=write_through_or_disable_on_unclassified_store_or_unknown_visibility
future_behavior_cvar=arm64_blue_dragon_fpscr_cfg_carrier_fastpath
```

The direct and tail-call ABI still pass only guest-return metadata in `x0`, so a
parent-to-callee fpscr payload must not be silently bolted onto normal calls.
Use either a callee-local carrier seeded from context loads, or return to the
caller-local `82282490:82282598 -> 82287788` edge-variant storage lane.

## Decision

No behavior patch yet. The counters prove dynamic fpscr traffic and exact call
writeback demand, but a speed A/B would still be meaningless because
`payload_materializations=0` by design. The next code patch should either:

- add a default-off `arm64_blue_dragon_fpscr_cfg_carrier_fastpath` skeleton with
  counters, normal-entry fallback, dirty-state materialization, and exact
  writebacks, or
- return to caller-local edge-variant storage for
  `82282490:82282598 -> 82287788`.

Do not run a quiet fpscr speed A/B before behavior materialization counters are
intentionally nonzero and all writebacks remain accounted.
