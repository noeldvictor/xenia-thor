## Shape

The block does four different kinds of work:

1. CR6 gate:
   - `cmpli crf6, r30, 0`
   - three CR6 byte stores
   - `context_barrier`
   - branch to `822825E0`

2. Stack/call setup:
   - loads `r31`, derives `r11`
   - derives `r3` from `r1 + 0x60`
   - derives `r4` from `r11 + r28`
   - calls `0x82274DB0`

3. Vector math/store path:
   - derives multiple stack pointers from one `r[1]` load
   - `lvlx` / `lvrx` / `lvx128`
   - `vspltw` lane broadcasts
   - three `vmaddfp` / HIR `mul_add`
   - three `stvewx` stores shaped as address align, lane extract, store

4. FPR/helper/branch tail:
   - `lfs`, `fmuls`, FPSCR update
   - call `0x82287788`
   - CR0 compare/branch
   - CR6 compare/branch toward `822825C8`

## Interpretation

This block is too mixed for another broad "cache r1 harder" attempt. The HIR
already reuses a single `r[1]` value for several stack-derived args inside the
block, and the two calls plus five context barriers are exactly the places where
state-cache rules become expensive.

The most promising generated-code questions from this block are narrower:

- How expensive is generic `stvewx` lowering? This block has three dynamic-hot
  `stvewx` shapes, each lowered through align/truncate/lane extract/store.
- How expensive are `vspltw` and lane `extract`/`splat` on A64? This block has
  six splats and nine extracts around three `vmaddfp` rows.
- Are the two callees (`82274DB0`, `82287788`) actually cheap in this route, or
  is `822824F0` only entry-hot because it gates expensive callee/body work?

## Decision

Do not patch `822824F0` yet from entry counts alone. The next useful change is
either:

1. Add lower-noise per-block body-time attribution for `82282490`, then compare
   `822824F0` against the static VMX-heavy blocks.
2. Add an A64 codegen audit for `stvewx` / `extract` / `splat` shapes in
   `82282490` so a later peephole is driven by emitted-code cost and hit count.

Keep `arm64_context_promotion_gpr_livein_r1`,
`arm64_context_promotion_gpr_local_slots`, `arm64_context_pinned_gpr_r1`, and
`arm64_context_pinned_gpr_r1_fallthrough` default-off.
