# 821CE028 f[1] Clobber Audit

Date: 2026-05-24 10:21 EDT

## Goal

Resolve the unknown-call blocker from the `82287788` `f[1]` carrier audit.
`82287788` calls `0x821CE028` at `82287854` and `82287ED4`; the prior audit
had to block 8 of 10 `f[1]` loads because it did not know whether this child
clobbers `f[1]`.

## Findings

`821CE028` has no `f[1]` access in OptHIR:

```text
function=821CE028 phase=OptHIR target_offset=296:f[1]
parsed_rows=620 calls=2 helper_calls=0 unknown_calls=2 target_loads=0 target_stores=0
decision=no_target_context_access_observed
```

The only calls in the filtered dump are the two return paths:

```text
call ordinal=612 pc=821CE1CC target=call_indirect.6 v752.i64<gpr0>
call ordinal=619 pc=821CE1D4 target=call_indirect.6 v754.i64<gpr0>
```

The hotpath report also confirms the only FPR context traffic in `821CE028` is
`f[0]`:

```text
Context Load Classes: FPR 1
Context Store Classes: FPR 1
Context Load Offsets: +288 f[0] 1
Context Store Offsets: +288 f[0] 1
```

There are no `+296` (`f[1]`) loads/stores and no `+2628` (`fpscr`) loads/stores
in this dump. The child is dominated by VMX/vector compare, CR stores, and
GPR/VMX state traffic instead.

## Decision

- Treat `821CE028` as `f[1]`-preserving for the narrow `82287788` carrier
  lane, because it does not access context offset `+296`.
- The two `call_indirect.6` rows are return paths, not nested child calls that
  should block `f[1]` preservation.
- This does not make `821CE028` a speed-patch target. It only removes the
  `f[1]` call-clobber uncertainty for the parent `82287788` carrier question.
- Keep `fpscr` out of the carrier lane. `82287788` itself still mutates fpscr,
  even though `821CE028` does not.
