# aX360e A64 Hard Swap

Date: 2026-05-18 16:41 EDT
Branch: `master`

## Question

The mini-JIT path was moving too slowly one operation at a time. The user asked
to stop polishing it and pull in the aX360e-style A64 backend because aX360e is
publicly presented as an Android Xbox 360 emulator based on Xenia.

## Sources Checked

- aX360e project page, accessed 2026-05-18:
  <https://aenu.cc/ax360e/>
  - The page describes aX360e as an Android Xbox 360 emulator ported from
    Xenia and links the public GitHub repository.
- `aenu1/ax360e` GitHub repository, accessed 2026-05-18:
  <https://github.com/aenu1/ax360e>
  - The repository describes itself as an experimental Android Xbox 360
    emulator based on Xenia and Xenia Canary.
- Local donor checkout:
  `scratch/upstream/ax360e/app/src/main/cpp/xenia-canary/src/xenia/cpu/backend/a64`

## Decision

Replace the active helper-backed `src/xenia/cpu/backend/arm64` mini-JIT with the
donor A64 backend family.

The import is still research-only. It is not a claim that this fork now has
working Android Xbox 360 compatibility. The immediate value is a real A64
backend shape: code cache, A64 emitter, backend context, stackpoint model,
native HIR sequence table, and donor transition assumptions.

## Imported Surface

- Copied donor backend files to `src/xenia/cpu/backend/a64`.
- Kept `src/xenia/cpu/backend/arm64/arm64_backend.h` as a thin compatibility
  alias to `a64::A64Backend`.
- Replaced `src/xenia/cpu/backend/arm64/premake5.lua` so the Android `arm64`
  project builds donor `a64` sources.
- Added donor `src/xenia/cpu/backend/code_cache_base.h`.
- Added donor ARM64 platform feature helpers:
  `src/xenia/base/platform_arm64.h` and `.cc`.
- Removed the old active mini-JIT implementation files.

## Compatibility Shims Added

- Mapped donor tracer logging to this fork's logging API.
- Mapped donor `DEFINE_int64` / `DECLARE_int64` uses to supported cvar types.
- Mapped Capstone `AARCH64` include/architecture names to this tree's `ARM64`
  Capstone API.
- Added small HIR helper methods expected by the donor backend:
  `GetOpcodeInfo`, `GetOpcodeNum`, `GuestAddressFor`, `UnpackOpcodeSig`, and
  `GetOpcodeName`.
- Registered donor-only HIR opcode IDs needed for the donor sequence tables:
  `DELAY_EXECUTION`, `VECTOR_DENORMFLUSH`, `TO_SINGLE`, `SET_NJM`, vector
  left/right load/store, and reserved load/store.
- Replaced C++20 `std::bit_cast` uses with this fork's C++17-safe
  `xe::memory::Reinterpret`.
- Used the reserved POSIX unwind buffer for normal Android instead of the
  aX360e-only `eh_frame_table_` scratch pointer.
- Stubbed donor XexModule info-cache MMIO recording until the matching
  XexModule info-cache import is done.

## Risks

- Runtime no longer dies immediately, but this is still not title-screen proof.
  The current wall moved to black-screen / draw-wait / first-present debugging.
- The donor info-cache path is not imported, so recorded-MMIO optimization is
  disabled for now.
- The Android app shell still has legacy debug-script flags in older docs and
  scripts, but the activity no longer forwards removed mini-JIT cvars.
- A64 ABI assumptions, backend context layout, stackpoint/unwind behavior, and
  HIR opcode semantics remain risk areas even though the current run survives
  initial game boot.
