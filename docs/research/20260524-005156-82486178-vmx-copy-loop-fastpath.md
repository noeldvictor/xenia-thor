  `bne`, and `rldicr`.

## Patch

Added a default-off, title/block-gated A64 experiment:

- Cvar: `arm64_blue_dragon_vmx_copy_loop_fastpath=false`.
- Android extra passthrough:
  `arm64_blue_dragon_vmx_copy_loop_fastpath`.
- Thor launch flag:
  `-Arm64BlueDragonVmxCopyLoopFastpath true|false`.
- Backend hook:
  `A64Emitter::TryEmitBlueDragonVmxCopyLoopBlock`.

The current implementation only fires for function `82486178`, block
`8248627C`, and replaces the inner loop with direct A64 loads/stores:

- load PPC `r30`, `r4`, `r31`, and `r28` once;
- loop over `(r30 + 4) & ~0xF` to `(r4 + 4) & ~0xF` with four 16-byte vector
  loads and stores per trip;
- update final `r28`, `r30`, `r4`, `r31`, final volatile GPR side effects, VMX
  registers `v0`, `v13`, `v12`, `v11`, `xer_ca`, and CR0.

This deliberately does not preserve per-iteration context stores or per-trip
block counter entries. It is a research-only shortcut, default-off, and must not
be promoted without a clean correctness and speed proof.

## Decision

Keep `arm64_blue_dragon_vmx_copy_loop_fastpath` default-off.

The fastpath is route-clean and shrinks generated code for this block, but the
same-APK inner-loop A/B did not prove speed. The control had lower cumulative
`82486178` and `82490030` body time in the final quiet route, so this is not a
promotion candidate.

Do not repeat the exact `82486178` fastpath A/B unchanged. The next useful speed
slice should use fresh evidence to inspect the remaining `82490030` children:
`82485DD8` and `82486018`, or rerun a low-noise comparator only if route drift
makes the previous A/B inconclusive.
