# A64 Block Profiler For 8272A3A4

Date: 2026-05-19 23:20 EDT

## Question

`8272A3A4` dominates filtered A64 body time in the Blue Dragon loading route.
Function-level counters are no longer enough: which HIR blocks inside that
guest function are hot enough to inspect first?

## Change

- Added default-off `arm64_speed_profile_block_filter`.
- The block filter matches guest function start addresses exactly, or explicit
  start-address ranges. A single address such as `8272A3A4` instruments only the
  function whose start is `8272A3A4`.
- Added per-block atomic counters and guest block-address labels to
  `A64Function`.
- The emitter inserts block-entry counters only when the block filter matches.
  Normal speed captures leave `current_a64_function_` null and do not allocate
  or clear block counter state.
- Routed the cvar through Android launch extras and `tools/thor/thor_xenia_debug.ps1`.

## Decision

Keep the block profiler, but treat it as a short OODA microscope:

- enable it only with a concrete exact function start;
- keep captures short enough to harvest the first useful interval;
- never call a block-profiler run faster or slower than a trace-off speed run;
- use it to choose the next PPC/Ghidra address queue.
