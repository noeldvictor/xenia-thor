## Interpretation

This explains why broad whole-function work is getting slow and noisy. The
current block profiler proves the early control/state blocks execute extremely
often, especially `822824F0`. It does not prove that the large static
VMX/vector blocks are the dynamic body-time leaders inside the opening route.

The next patch should not repeat the `r[1]` live-in A/B unchanged and should not
restart broad VMX/dot/permute work. The best next choices are:

1. Target dynamic hot block `822824F0` with a narrow tool/code-size audit. It
   mixes CR6 stores, stack GPR traffic, calls, vector loads, `vspltw`,
   `vmaddfp`, and `stvewx`, so it is more representative than a single GPR
   cache attempt.
2. Add a lower-noise block body-time profiler before touching the static
   vector-heavy blocks. Entry counts alone may over-rank early control blocks,
   but static HIR size alone may over-rank cold vector blocks.
